# Projects & Memory 활용법

> 매번 같은 설명을 반복하지 마세요. 프로젝트를 만들면 Claude가 "우리 팀 규칙"을 기억하고, Memory 기능으로 대화할수록 더 똑똑해집니다.

## Projects란?

프로젝트(Projects)는 **주제별로 Claude와의 작업 공간을 분리하는 기능**입니다.

| 프로젝트 없이 | 프로젝트 사용 시 |
|---|---|
| 매번 "보고서는 존댓말로 써줘" 반복 | 한 번 설정하면 자동 적용 |
| 모든 대화가 한 곳에 섞임 | 주제별로 깔끔하게 분리 |
| 이전 맥락을 매번 설명 | 프로젝트 안에서 맥락 유지 |
| 팀원과 공유 불가 | 팀원과 프로젝트 공유 가능 |

::: info 쉽게 이해하기
프로젝트 = **"Claude 전용 업무 폴더"** 라고 생각하면 됩니다. 이메일 작업은 "이메일" 폴더에, 보고서 작업은 "보고서" 폴더에 넣는 것처럼요.
:::

## 프로젝트 만들기 & 관리

### 새 프로젝트 만들기

1. Claude Desktop 좌측 사이드바에서 **"Projects"** 클릭
2. **"+ New Project"** 버튼 클릭
3. 프로젝트 이름 입력 (예: `주간보고`, `거래처 관리`)
4. 필요 시 설명(Description) 추가

### 프로젝트 이름 규칙 추천

| 분류 | 이름 예시 | 설명 |
|---|---|---|
| 정기 업무 | `주간보고`, `월간실적` | 반복 업무별로 분리 |
| 부서/팀 | `마케팅팀 콘텐츠`, `인사팀 채용` | 부서 업무별로 분리 |
| 프로젝트 | `2026 신제품 론칭`, `사옥 이전` | 한시적 프로젝트별로 분리 |
| 거래처 | `A사 커뮤니케이션`, `B사 제안서` | 거래처별로 분리 |

::: tip 이름 짓기 팁
프로젝트 이름은 **짧고 명확하게** 지어야 나중에 빨리 찾을 수 있습니다. "김대리가 부탁한 그 건"보다 "2026 상반기 예산안"이 훨씬 낫습니다.
:::

### 프로젝트 관리

- **이름 변경**: 프로젝트 설정(⚙️)에서 이름 수정 가능
- **삭제**: 프로젝트 설정에서 삭제 (삭제 시 안의 대화도 함께 삭제되니 주의)
- **보관(Archive)**: 당장 쓰지 않지만 삭제하긴 아까운 프로젝트는 보관 처리

::: warning 주의
프로젝트를 삭제하면 **안에 포함된 모든 대화 기록이 함께 삭제**됩니다. 중요한 결과물은 반드시 별도로 저장한 뒤 삭제하세요.
:::

## Custom Instructions 설정

Custom Instructions는 프로젝트의 **핵심 기능**입니다. 프로젝트 안에서 Claude가 항상 따라야 할 규칙을 미리 정해두는 것이죠.

### 설정 방법

1. 프로젝트 열기
2. 프로젝트 이름 옆 **설정(⚙️) 아이콘** 클릭
3. **"Custom Instructions"** 영역에 규칙 입력
4. 저장

### 실전 예시: 보고서 작성 규칙

```text
이 프로젝트에서는 아래 규칙을 항상 따라줘:

1. 문체: 존댓말(합니다체) 사용
2. 분량: A4 기준 2장 이내
3. 구조: 반드시 표(table) 포함
4. 머리말: "OO팀 주간 업무 보고" 제목으로 시작
5. 날짜: 보고 기간을 상단에 표시 (예: 2026.03.09 ~ 2026.03.13)
6. 마무리: "이슈사항" → "다음 주 계획" 순서로 끝내기
```

### 실전 예시: 브랜드 톤 & 회사 정보

```text
우리 회사 기본 정보:
- 회사명: (주)한빛솔루션
- 영문명: Hanbit Solutions Inc.
- 대표: 박진우
- 업종: IT 컨설팅 / 디지털 전환 솔루션
- 슬로건: "기술로 밝히는 내일"

브랜드 톤:
- 전문적이되 딱딱하지 않게
- 자신감 있되 겸손하게
- "파트너"라는 단어를 자주 사용
- "솔루션 제공"보다 "함께 해결"이라는 표현 선호
- 외래어 남용 금지 (가능하면 한국어 표현 사용)
```

### 실전 예시: 이메일 커뮤니케이션 규칙

```text
이메일 작성 시 규칙:
- 서두: 인사말 포함 ("안녕하세요, OOO입니다.")
- 본문: 핵심을 먼저, 배경 설명은 그 다음
- 요청사항: 번호 매기기로 명확하게
- 마무리: "검토 부탁드립니다. 감사합니다."
- 서명: 내 이름/직책/연락처 포함
- 회신 기한이 있으면 반드시 명시

내 정보:
- 이름: 김서연
- 직책: 마케팅팀 과장
- 연락처: 02-1234-5678 (내선 302)
- 이메일: sy.kim@hanbit.co.kr
```

::: tip Custom Instructions를 잘 쓰는 비결
**"매번 반복해서 말하는 것"을 적으세요.** 대화할 때 "아, 또 이거 설명해야 하네" 싶은 내용이 바로 Custom Instructions에 넣어야 할 내용입니다.
:::

## Memory 기능

Memory는 Claude가 **여러 대화에 걸쳐 배운 내용을 기억하는 기능**입니다. Custom Instructions가 "내가 직접 알려준 규칙"이라면, Memory는 "Claude가 대화하면서 스스로 학습한 내용"입니다.

### Memory가 기억하는 것

| 기억함 ✅ | 기억하지 않음 ❌ |
|---|---|
| 선호하는 문체/톤 | 일회성 대화 내용 |
| 자주 사용하는 양식 | 민감한 개인정보 (비밀번호 등) |
| 사용자의 역할/직책 | 특정 대화의 세부 내용 전체 |
| 피드백으로 교정한 내용 | 다른 사용자의 정보 |
| 반복적으로 요청한 스타일 | 한 번만 언급된 임시 정보 |

### Memory가 쌓이는 과정

```text
[1차 대화]
나: "보고서 써줘" → Claude가 기본 스타일로 작성
나: "아, 우리 팀은 불릿 포인트 대신 번호 매기기를 써"

[2차 대화]
나: "보고서 써줘" → Claude가 번호 매기기 스타일로 작성 ✅
(이전에 배운 선호사항을 기억)
```

### 기억 확인 & 관리

**확인 방법:**
1. Claude Desktop 설정(Settings)으로 이동
2. **"Memory"** 섹션 확인
3. Claude가 기억하고 있는 항목 목록 확인

**특정 기억 삭제:**
- 기억 목록에서 개별 항목 옆 **삭제 버튼** 클릭

**전체 기억 초기화:**
- Memory 설정에서 **"Clear All Memory"** 선택

**대화 중 관리:**
```text
지금부터 보고서 양식은 이전 스타일 말고,
표 중심으로 바꿔서 기억해줘.
```

```text
내가 전에 "반말로 써달라"고 한 거 잊어줘.
앞으로는 존댓말로 작성해줘.
```

::: warning Memory 관리 팁
Claude가 잘못 기억한 정보가 있으면 빨리 수정하세요. 잘못된 기억이 누적되면 이후 모든 대화에 영향을 줍니다. 정기적으로 Memory 목록을 확인하는 습관을 들이세요.
:::

## 프로젝트 활용 시나리오

### 시나리오 1: "주간보고" 프로젝트

**목표:** 매주 같은 양식으로 보고서를 빠르게 작성

**Custom Instructions 설정:**
```text
주간 업무 보고서 규칙:
- 제목: "[마케팅팀] 주간 업무 보고 (MM.DD ~ MM.DD)"
- 구조: 핵심 성과(3개) → 진행 중(표) → 이슈 → 다음 주 계획
- 성과는 수치(%, 건, 원)를 반드시 포함
- 전체 분량 A4 1.5장 이내
- 보고 대상: 김 팀장님
```

**매주 이렇게만 입력하면 끝:**
```text
이번 주(3/9~3/13) 보고서 써줘:

한 일:
- 인스타 캠페인 집행, 도달 12만
- 블로그 포스팅 3건 발행
- A사 콜라보 미팅

이슈:
- 예산 소진율 87%, 추가 배정 필요

다음 주:
- 2분기 캠페인 기획안 제출
- 유튜브 채널 리뉴얼 시안 검토
```

### 시나리오 2: "거래처 관리" 프로젝트

**목표:** 거래처별 히스토리와 커뮤니케이션을 체계적으로 관리

**Custom Instructions 설정:**
```text
거래처 커뮤니케이션 규칙:
- 항상 "귀사"가 아닌 회사명으로 지칭
- 이전 미팅/통화 내용을 요약에 포함
- 후속 조치(Action Item)를 반드시 목록으로 정리
- 이메일은 항상 격식체, CC 대상 제안 포함

주요 거래처:
- A사 (담당: 이 과장) - SI 프로젝트 진행 중
- B사 (담당: 박 대리) - 유지보수 계약
- C사 (담당: 최 차장) - 신규 제안 단계
```

**활용 예시:**
```text
오늘 A사 이 과장님과 통화한 내용 정리해줘:

- 3차 테스트 결과 공유받음, 2건 버그 발견
- 버그 수정 후 다음 주 화요일까지 재배포 요청
- 4월 1일 오픈 일정은 유지
- 추가 교육 일정 잡아달라고 함

이 내용으로:
1. 통화 요약 메모
2. A사에 보낼 확인 이메일
3. 우리 팀 내부 공유용 요약
이 3개를 만들어줘.
```

### 시나리오 3: "팀 온보딩" 프로젝트

**목표:** 신입/이동 팀원의 적응을 돕는 교육 자료 관리

**Custom Instructions 설정:**
```text
온보딩 자료 작성 규칙:
- 대상: IT 비전공자도 이해할 수 있는 수준
- 문체: 친근하고 편안한 말투 (~해요, ~이에요)
- 전문 용어는 반드시 괄호 안에 쉬운 설명 추가
- 단계별 설명 시 스크린샷 위치 표시 (예: [스크린샷: 로그인 화면])
- 체크리스트 형태 활용

우리 팀 정보:
- 팀명: 디지털혁신팀 (15명)
- 사용 툴: Slack, Notion, Jira, Google Workspace
- 근무 시간: 09:00~18:00 (유연근무제)
```

**활용 예시:**
```text
신입 팀원용 "첫 주 가이드"를 만들어줘.

Day 1~5로 나눠서:
- Day 1: 계정 설정, 자리 배치, 인사
- Day 2: 주요 툴 사용법 (Slack, Notion)
- Day 3: 진행 중인 프로젝트 브리핑
- Day 4: 실습 과제 (간단한 Notion 문서 작성)
- Day 5: 1:1 미팅, 질문 정리

각 날짜별로 체크리스트와 참고 링크 넣어줘.
```

## 팀 프로젝트 공유

팀원과 프로젝트를 공유하면 **같은 Custom Instructions 아래에서 일관된 결과**를 얻을 수 있습니다.

### 공유 방법

1. 공유할 프로젝트의 **설정(⚙️)** 클릭
2. **"Share"** 또는 **"Manage Members"** 선택
3. 팀원의 이메일 주소 입력
4. 권한 설정 (보기/편집)
5. 초대 전송

### 공유 시 권한 구분

| 권한 | 할 수 있는 것 | 할 수 없는 것 |
|---|---|---|
| **뷰어(Viewer)** | 프로젝트 내 대화, Instructions 확인 | Instructions 수정, 프로젝트 삭제 |
| **편집자(Editor)** | Instructions 수정, 대화 생성 | 프로젝트 삭제, 멤버 관리 |
| **소유자(Owner)** | 모든 권한 | - |

::: tip 팀 공유 활용 아이디어
- **팀 공통 프로젝트**: "마케팅팀 공통" 프로젝트에 브랜드 가이드라인을 Instructions로 설정 → 누가 작성해도 일관된 톤
- **인수인계 프로젝트**: 담당자 변경 시 프로젝트를 공유하면 맥락 전달이 쉬움
- **교육용 프로젝트**: 프롬프트 잘 쓰는 팀원의 프로젝트를 공유해 다른 팀원 학습에 활용
:::

## 프로젝트 정리 팁

### 프로젝트 구조 설계

::: info 추천 프로젝트 구조 (사무직 기준)
```
📁 정기 업무
  ├─ 주간보고
  ├─ 월간실적
  └─ 분기 리뷰

📁 커뮤니케이션
  ├─ 이메일 작성
  ├─ 거래처 A사
  └─ 거래처 B사

📁 현재 프로젝트
  ├─ 2026 신제품 론칭
  └─ 사내시스템 개편

📁 팀 운영
  ├─ 팀 온보딩
  └─ 회의록 정리
```
:::

### 프로젝트 정리 체크리스트

| 주기 | 할 일 |
|---|---|
| **매주** | 완료된 대화 정리 / 결과물 별도 저장 |
| **매월** | 안 쓰는 프로젝트 보관(Archive) 처리 |
| **분기별** | Custom Instructions 업데이트 / 프로젝트 구조 재점검 |
| **반기별** | 보관된 프로젝트 삭제 여부 검토 |

### Custom Instructions 관리 요령

```text
[좋은 예 - 구체적이고 구조화]
보고서 규칙:
1. 문체: 존댓말
2. 분량: A4 2장
3. 필수 포함: 표, 수치
4. 보고 대상: 김 팀장

[나쁜 예 - 모호하고 너무 김]
보고서 잘 써줘. 깔끔하게.
너무 길지 않게. 적당히.
```

::: tip 프로젝트를 너무 잘게 나누지 마세요
프로젝트가 20개가 넘어가면 오히려 찾기 어렵습니다. **"하나의 업무 맥락 = 하나의 프로젝트"** 원칙으로, 5~10개 수준이 관리하기 가장 좋습니다.
:::

::: warning 보안 주의사항
Custom Instructions에 **비밀번호, API 키, 개인정보(주민번호 등)** 를 절대 입력하지 마세요. Instructions는 프로젝트를 공유할 때 다른 사람에게도 보입니다.
:::
